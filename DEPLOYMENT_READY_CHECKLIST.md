**Deployment Cookies & CSRF**

- **Purpose:** Ensure session and CSRF cookies are accepted when clients access the site via different origins (e.g., `localhost`, `127.0.0.1`, or LAN IP like `192.168.x.x`). Modern browsers require special cookie settings when cookies must be sent cross-site.
- **Env vars (recommended):**
  - `SESSION_COOKIE_SAMESITE`: set to `None` in production to allow cross-site cookies. Example: `SESSION_COOKIE_SAMESITE=None`.
  - `SESSION_COOKIE_SECURE`: set to `True` in production (required when `SameSite=None`). Example: `SESSION_COOKIE_SECURE=True`.
  - `REMEMBER_COOKIE_SAMESITE` / `REMEMBER_COOKIE_SECURE`: same rules as session cookie for "remember me" cookies.
  - `SESSION_COOKIE_DOMAIN` (optional): set to a domain if you need the cookie shared across subdomains.

- **Important (HTTPS requirement):**
  - Browsers accept cookies with `SameSite=None` only when `Secure=True`, which requires HTTPS. For production deployments that need cross-origin cookie sharing, ensure TLS is enabled (via reverse proxy, load balancer, or managed certificates).

- **Local / CI testing guidance:**
  - Keep `SESSION_COOKIE_SAMESITE=Lax` and `SESSION_COOKIE_SECURE=False` in `Development` and `Testing` to avoid needing HTTPS locally.
  - Tests already use `TestingConfig` which disables CSRF and sets non-secure cookies; no test changes required for local runs.

- **Example PowerShell to run production-like server (with env overrides):**
  ```powershell
  $Env:FLASK_ENV='production'
  $Env:SESSION_COOKIE_SAMESITE='None'
  $Env:SESSION_COOKIE_SECURE='True'
  $Env:REMEMBER_COOKIE_SAMESITE='None'
  $Env:REMEMBER_COOKIE_SECURE='True'
  .\venv\Scripts\python.exe .\run.py
  ```
